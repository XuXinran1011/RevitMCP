# 贡献指南

感谢您对RevitMCP项目的关注！本指南将帮助您开始为项目做出贡献。

## 入门

1. **Fork仓库**
   - 在GitHub上创建自己的仓库副本

2. **克隆您的Fork**

   ```bash
   git clone https://github.com/您的用户名/RevitMCP.git
   cd RevitMCP
   ```

3. **设置开发环境**
   - 安装所需依赖项：
     - Windows 10或更高版本
     - Autodesk Revit 2025
     - .NET 8 / .NET 9
     - Visual Studio 2022或更高版本（推荐）
     - Claude Desktop（用于测试）

4. **创建分支**

   ```bash
   git checkout -b feature/您的功能名称
   ```

## 开发指南

### 设计原则与模式

请确保您的代码遵循项目的核心设计原则：

- **关注点分离** - 每个类和模块应该只有一个责任
- **SOLID原则** - 特别注意单一职责和依赖倒置原则
- **可测试性** - 编写易于测试的代码，使用依赖注入
- **安全第一** - 始终验证输入并安全处理异常

在实现新功能时，考虑使用项目采用的设计模式：

- **依赖注入** - 通过构造函数注入依赖，而不是直接创建
- **适配器模式** - 为Revit API创建适配器，隔离外部依赖
- **命令模式** - 封装操作为命令对象
- **工厂模式** - 使用工厂创建复杂对象

详细信息请参阅[ARCHITECTURE.md](ARCHITECTURE.md#设计原则与模式)文档。

### 编码标准

- 遵循C#编码约定
- 使用有意义的变量和函数名
- 编写清晰的注释和文档
- 为公共API包含XML文档
- 遵循现有的项目结构和模式

### 测试

- 为新功能编写单元测试
- 在提交拉取请求前确保所有测试通过
- 使用Revit 2025测试您的更改
- 使用Claude Desktop或其他MCP客户端验证功能

### 提交消息

- 使用清晰、描述性的提交消息
- 在适用时引用问题编号
- 遵循格式：`[组件] 更改的简要描述`

## 拉取请求流程

1. **更新文档**
   - 更新相关文档以反映您的更改
   - 在必要时为代码添加注释

2. **提交拉取请求**
   - 将您的更改推送到您的fork
   - 向主仓库提交拉取请求
   - 提供对更改及其目的的清晰描述
   - 引用任何相关问题

3. **代码审查**
   - 处理代码审查中的反馈
   - 进行必要的更改并更新您的分支

4. **合并**
   - 一旦获得批准，您的拉取请求将被合并
   - 庆祝您的贡献！

## 沟通

- 使用GitHub Issues报告错误和功能请求
- 加入我们的社区讨论进行一般性问题
- 在所有沟通中遵循[行为准则](CODE_OF_CONDUCT.md)

## 开发工作流

### 使用MCP SDK

RevitMCP项目使用官方MCP C# SDK。在使用SDK时：

1. 参考[modelcontextprotocol.io](https://modelcontextprotocol.io/)的官方文档
2. 使用SDK内置的工具进行调试和测试
3. 遵循SDK的实现最佳实践

### 使用Revit API

在与Revit API交互时：

1. 遵循Autodesk的API使用指南
2. 使用不同的Revit模型进行彻底测试
3. 适当处理异常和边缘情况
4. 记录任何Revit特定的行为或限制
